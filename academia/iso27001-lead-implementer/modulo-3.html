Definicion
Que es un GAP Analysis
Un GAP Analysis consiste en un analisis de cumplimiento tanto de los requisitos de ISO/IEC 27001 como de sus controles del Anexo A. Es la herramienta que permite identificar la diferencia entre el estado actual de la organizacion y los requisitos que debe cumplir para obtener la certificacion.
“El GAP Analysis responde a tres preguntas: donde estamos, donde debemos estar, y que nos falta para llegar.”
Metodologia CertiProf
Sin un GAP Analysis riguroso, cualquier plan de implementacion se construye sobre suposiciones. El resultado: plazos irreales, recursos insuficientes y sorpresas desagradables durante la auditoria.
Objetivos
Para que sirve el GAP Analysis
1Identificar las brechas entre el estado actual y los requisitos de ISO 27001
2Priorizar las acciones necesarias para alcanzar la conformidad
3Estimar el esfuerzo, tiempo y recursos necesarios para la implementacion
4Establecer una linea base para medir el progreso durante el proyecto
5Proporcionar a la direccion informacion objetiva para la toma de decisiones
Metrica
Modelos de madurez: como medimos
Para realizar el GAP Analysis, es recomendable utilizar un modelo de madurez que permita evaluar el cumplimiento de forma objetiva. Los modelos mas comunes incluyen:
COBITModelo de madurez de ISACA, ampliamente utilizado para gobierno y gestion de TI
NISTFramework de ciberseguridad con niveles de implementacion progresivos
ISM3Modelo de madurez de seguridad de la informacion especifico
SSE-CMMCapability Maturity Model para ingenieria de seguridad de sistemas
Escala
Niveles de madurez COBIT
0Inexistente. La gestion de procesos no se aplica en absoluto
1Inicial / Ad-hoc. Los procesos son desorganizados y reactivos
2Repetible. Los procesos siguen un patron regular pero no estandarizado
3Definido. Los procesos estan documentados y comunicados
4Gestionado / Medible. Los procesos son monitorizados y medidos
5Optimizado. Las buenas practicas son utilizadas y automatizadas, con mejora continua
Este modelo se adapta perfectamente para establecer un nivel de madurez actual y un nivel objetivo para cada uno de los requisitos (clausulas) y controles (Anexo A) de ISO/IEC 27001.
Metodologia
Como realizar un GAP Analysis
El proceso tipico para realizar un GAP Analysis incluye los siguientes pasos:
1Definir el alcance. Determinar que areas, procesos y ubicaciones se incluyen en el analisis
2Seleccionar el modelo de madurez. Elegir la escala y criterios de evaluacion
3Recolectar evidencias. Revisar documentacion existente, entrevistar personal, observar procesos
4Evaluar cada requisito. Asignar un nivel de madurez actual a cada clausula y control relevante
5Identificar brechas. Comparar el nivel actual con el nivel requerido para la certificacion
6Documentar hallazgos y recomendar acciones. Priorizar las brechas segun su criticidad
Error tipico
El GAP Analysis que no es un analisis
“Ya sabemos lo que nos falta. No hace falta un analisis formal.”
Frase que precede a un proyecto con sobrecostes y plazos incumplidos.
El error mas frecuente es saltarse el GAP Analysis o hacerlo superficialmente, asumiendo que "se conoce" el estado de la organizacion. Un GAP Analysis basado en suposiciones, sin evidencias objetivas, no es un analisis — es una opinion documentada. Y las opiniones no sirven para planificar un proyecto de implementacion.
❌Enfoque incorrecto: "Cumplimos la mayoria de los requisitos. Solo falta documentar algunas politicas."
✅Enfoque correcto: "Tras evaluar 87 controles con evidencias objetivas, tenemos un cumplimiento del 42% en categoria Media y del 18% en categoria Alta."

APLICACIÓN PRÁCTICA

Cómo llevar el contenido del módulo a una organización real

Este módulo debe utilizarse dentro de diseño, implantación, operación y mejora de un SGSI basado en riesgos. Su valor no está en repetir definiciones, sino en decidir qué debe ocurrir, quién participa y cómo se demuestra.

Para trabajar «GAP Analysis: donde estamos y donde debemos estar», comienza delimitando alcance, criterio, responsables, operación, evidencias y mejora. Después conecta el requisito o principio con los procesos, sistemas, terceros y decisiones que realmente existen en la organización.

Los actores habituales serán Dirección, Seguridad, Riesgos, Tecnología, Legal, Personas y propietarios de activos. No todos ejecutan el control, pero cada uno puede aportar una parte del contexto, la evidencia o la decisión.

  1. Define el objeto y el alcance antes de discutir controles.
  2. Identifica propietario, ejecutor, revisor y autoridad de excepción.
  3. Distingue diseño, implantación, operación y eficacia.
  4. Comprueba cambios, incidencias y excepciones, no solo el estado ideal.
  5. Cierra con una decisión y una acción verificable.
EXPEDIENTE DEL MÓDULO

Documentación y evidencia que debería existir

La documentación no debería ser una colección de archivos aislados. Debe permitir seguir una relación lógica entre criterio, decisión, actividad y resultado.

Cada evidencia necesita contexto: sistema o proceso al que pertenece, periodo cubierto, fuente, responsable y estado. Sin esos datos, una captura o un documento pueden ser auténticos pero insuficientes.

  • alcance y contexto aprobados
  • metodología y evaluación de riesgos
  • plan de tratamiento y Declaración de Aplicabilidad
  • procedimientos, registros y métricas operativas
  • auditoría interna, revisión y acciones de mejora
CASO DE REVISIÓN

Aplicación de «GAP Analysis: donde estamos y donde debemos estar»

Escenario: una organización quiere implantar ISO 27001 y comienza por redactar políticas sin haber definido alcance, contexto ni riesgos. El objetivo no es memorizar una respuesta, sino reconstruir qué debería verificarse y con qué evidencia.

1 · Delimitar

Define el alcance y concreta alcance, criterio, responsables, operación, evidencias y mejora.

2 · Corroborar

Contrasta entrevistas con documentos, registros y observación de la operación.

3 · Concluir

Separa hechos, criterio aplicable, impacto y acción necesaria.

ERRORES FRECUENTES

Dónde suele fallar la aplicación

  • implantar controles antes de entender el riesgo
  • copiar documentación genérica
  • dar por terminado el SGSI al obtener la certificación
  • tratar una excepción permanente como si fuera temporal
  • cerrar acciones por actividad realizada, sin validar su eficacia
COMPROBACIÓN

Preguntas que deberías poder responder

  • ¿Qué decisión concreta cambia después de comprender «GAP Analysis: donde estamos y donde debemos estar»?
  • ¿Qué evidencia pedirías primero y qué necesitarías corroborar?
  • ¿Qué parte depende del diseño y cuál de la operación diaria?
  • ¿Qué excepción sería aceptable y quién tendría autoridad para aprobarla?
  • ¿Cómo comprobarías dentro de tres meses que sigue funcionando?
EJEMPLO GUIADO

Caso práctico: GAP Analysis: donde estamos y donde debemos estar

Una consultora usa una checklist genérica de 93 controles y concluye que tiene un 72 % de madurez, sin revisar contexto, riesgos ni evidencias.

CONTEXTO TÉCNICO

M365, VPN, repositorios, portátiles y proveedores.

DECISIÓN

  • Realizar GAP contra requisitos y controles aplicables, con evidencia y prioridad.
  • Asignar un responsable y una fecha de revisión.
  • Definir cómo se comprobará la eficacia.

EVIDENCIA

  • Estado actual, evidencia, brecha, riesgo, esfuerzo, propietario y dependencia.
  • Corroborar con una fuente independiente.
  • Conservar población, periodo y responsable.

ERROR A EVITAR

  • Convertir el GAP en una puntuación sin trazabilidad.
  • Aceptar afirmaciones sin prueba suficiente.
  • Cerrar acciones sin verificación posterior.
